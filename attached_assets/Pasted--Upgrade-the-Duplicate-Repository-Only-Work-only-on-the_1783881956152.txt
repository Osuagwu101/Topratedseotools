## 3. Customer Testimonials

Create a professional customer-testimonial section.

Because the website is new, do not create or publish invented testimonials as genuine customer experiences.

Instead:

* Add sample testimonial templates inside the Admin Dashboard only.
* Clearly label each template: “Sample Content – Edit Before Publishing.”
* Keep all sample testimonials unpublished by default.
* Do not display them publicly until the administrator edits and manually approves them.
* Do not describe sample content as verified.

The Admin Dashboard must allow the administrator to:

* Add a testimonial
* Edit a testimonial
* Preview it

* Delete it
* Reorder testimonials
* Add the customer’s display name
* Add an optional photograph or avatar

Add safe sample templates such as:

Do not add any screenshot request, screenshot upload, access-delivery screenshot, or screenshot placeholder feature to the testimonial section.

## 4. Verified Purchase Reviews

Build a genuine verified-purchase review system connected to completed orders.

A customer should only receive a “Verified Purchase” badge when all the following conditions are met:

* The review belongs to a registered customer account.
* The customer has a successfully paid and completed order.
* The reviewed product is included in that completed order.
* The order has not been refunded, cancelled, reversed, disputed, marked as fraudulent, or invalidated.

### Review Prompt Behaviour

After an eligible completed purchase:

* Ask the customer to review the purchased product the next time they log in or return while authenticated.
* Show the review request no more than once per login session.
* Show the reminder during a maximum of three separate login sessions.
* Do not count page refreshes or navigation between pages as new login sessions.
* Allow the customer to submit the review, dismiss the current reminder, or choose “Review later.”
* After the third ignored or dismissed login prompt, stop displaying the reminder for that purchase.
* Reset the review-prompt process only when the customer renews the product or completes another eligible purchase.
* Never show another review prompt for an order or product that has already been reviewed.
* Where several products are purchased in one order, provide a simple interface where each eligible product can be reviewed separately.

The review prompt should be presented as a clean modal or dashboard notification. It must have a visible close or postpone option and must not repeatedly interrupt the customer during the same session.

### Review Fields

Include:

* Star rating from 1 to 5
* Written review
* Optional review title
* Date submitted
* Customer display name or anonymised display format
* Product reviewed
* Verified-purchase status

### Admin Review Moderation

Add review moderation tools to the Admin Dashboard that allow administrators to:

* View pending, approved, rejected, hidden, and reported reviews
* Approve or reject reviews
* Hide or restore reviews
* Reply publicly to reviews
* Remove abusive, irrelevant, fraudulent, misleading, or privacy-violating content
* Search and filter by product, rating, status, customer, and date
* Privately view the related order for verification

The administrator must not be able to falsely apply the Verified Purchase badge.

Do not allow administrators to change the meaning of a customer’s review. Where moderation is required, allow the review to be hidden or rejected instead of silently rewriting the customer’s words.

### Public Review Display

Display only approved reviews.

Show:

* Average product rating
* Total number of approved reviews
* Rating breakdown
* Verified Purchase badge where applicable
* Review date
* Product reviewed
* Administrative response, where one exists

## 5. Customers Served Counter

Add a “Customers Served” counter in a suitable trust section of the website.

### Initial Configuration

* Set the starting baseline to 100.
* The next valid qualifying purchase should display 101.
* Increase the counter only after a genuinely successful and confirmed payment.
* Do not increase it for failed, pending, abandoned, test, cancelled, refunded, reversed, disputed, duplicated, or fraudulent transactions.

Store the baseline and live qualifying count separately.

Use the following calculation:

Displayed customers served = administrator-approved baseline + qualifying completed customers or orders after launch.

### Counting Options

Add an Admin Dashboard setting that allows the administrator to choose between:

1. **Unique Customers Served:** Each eligible customer is counted only once.
2. **Successful Orders Completed:** Every valid completed order increases the count by one.

Use **Unique Customers Served** as the default.

A customer purchasing several products in one transaction should increase the counter by one under the Unique Customers Served setting.

The Admin Dashboard should show:

* Starting baseline
* New verified customers or orders counted
* Current displayed total
* Counting method
* Last updated date
* Protected manual correction option
* Audit history of manual corrections

Do not display phrases such as “over,” “more than,” “thousands,” or similar wording unless the verified number genuinely supports the statement.

## 6. Accepted Payment Method Icons

Add a clean payment-method area showing only the icons of payment methods genuinely accepted through the website’s active Paystack checkout.

Do not display payment-method names, descriptions, explanatory sentences, cards, labels, headings beneath individual icons, or promotional text beside the icons.

The frontend payment display should contain icons only.

Examples may include:

* Visa
* Mastercard
* Verve
* Bank transfer
* OPay
* Other payment methods genuinely supported and enabled through Paystack

Only display an icon when that payment method is genuinely available and functional at checkout.

Do not display OPay or any other payment method merely because an icon exists. Confirm that it is supported by the website’s active Paystack configuration before making it visible.

### Display Requirements

* Use clean, recognisable, official-looking payment icons.
* Keep all icons visually consistent.
* Use consistent sizing, spacing, alignment, and background treatment.
* Make the icon row responsive on desktop, tablet, and mobile.
* Allow icons to wrap cleanly on smaller screens.
* Add accessible alt text or screen-reader labels even though visible text should not appear beside the icons.
* Optimise the icons so that they do not slow down the website.
* Avoid oversized icons or a large payment section that overcrowds the homepage.
* Do not display payment methods that are unavailable for the customer’s country, currency, account, or checkout configuration where this can be determined.

### Admin Controls

Create controls in the Admin Dashboard that allow the administrator to:

* Enable or disable the entire payment-icon section
* Add a supported payment method
* Remove a payment method
* Enable or disable individual icons
* Reorder the icons
* Preview the icon row before publishing
* Upload or select an appropriate icon
* Add accessible alt text
* Mark the active payment provider
* View whether a method is manually configured or automatically detected

Where technically possible, automatically synchronise the displayed payment methods with the methods enabled through Paystack.

Do not let frontend icon settings override the actual Paystack checkout configuration. The checkout provider remains the source of truth.
